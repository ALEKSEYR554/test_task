# Трекер задач — модуль МИС

Многопользовательский трекер задач с тегами, календарём и обработкой периодических задач для медицинской информационной системы.

## Возможности

- **Многопользовательская авторизация**: администратор создаёт пользователей, вход/выход
- **CRUD задач**: создание, просмотр, редактирование, удаление с фильтрацией по дате и статусу
- **Система тегов**: глобальные обязательные теги + пользовательские теги с цветами
- **Календарь**: переключение неделя/месяц, цветовая кодировка по тегам, детальный вид дня
- **Периодические задачи**: ежедневные, ежемесячные, на конкретные даты, чётные/нечётные дни
- **API**: RESTful JSON эндпоинты с Swagger документацией

## Запуск без Docker

### Требования

- Ruby 3.3.5+
- PostgreSQL 16+
- Bundler

### Установка

1. Клонировать репозиторий:
```bash
git clone <url-репозитория>
cd test_task
```

2. Установить зависимости:
```bash
bundle install
```

3. Настроить базу данных — отредактировать `config/database.yml`:
```yaml
development:
  <<: *default
  database: test_task_development
  username: postgres
  password: ваш_пароль
  host: localhost
```

4. Создать и заполнить базу данных:
```bash
bin/rails db:create
bin/rails db:migrate
bin/rails db:seed
```

5. Запустить сервер:
```bash
bin/rails server
```

6. Открыть `http://localhost:3000`

## API Документация

Swagger документация: `http://localhost:3000/api-docs`

### Эндпоинты

#### Авторизация
- `POST /api/v1/auth/token` — получить токен (email + password)

#### Задачи
- `GET /api/v1/tasks` — список задач (фильтр по дате и статусу)
- `GET /api/v1/tasks/:id` — получить задачу
- `POST /api/v1/tasks` — создать задачу
- `PUT /api/v1/tasks/:id` — обновить задачу
- `DELETE /api/v1/tasks/:id` — удалить задачу

#### Теги
- `GET /api/v1/tags` — список тегов
- `GET /api/v1/tags/:id` — получить тег
- `POST /api/v1/tags` — создать тег
- `PUT /api/v1/tags/:id` — обновить тег (нельзя для обязательных)
- `DELETE /api/v1/tags/:id` — удалить тег (нельзя для обязательных)

### Авторизация API

1. Вызвать `POST /api/v1/auth/token` с body `{"email": "...", "password": "..."}`
2. Получить `auth_token` из ответа
3. Добавить заголовок ко всем запросам:
```
Authorization: Bearer <auth_token>
```

Или использовать Swagger UI: нажать "Authorize" и ввести `Bearer <токен>`.

## Схема базы данных

| Таблица | Описание |
|---------|----------|
| `users` | Пользователи (id, name, email, password_digest, role, auth_token) |
| `tasks` | Задачи (id, user_id, title, description, due_date, status, task_type, periodicity_type, periodicity_config) |
| `tags` | Теги (id, user_id, name, color, is_required) |
| `task_tags` | Связь задач-тегов (task_id, tag_id) |
| `periodic_exceptions` | Исключения периодических задач (task_id, date, status, one_time_task_id) |

## Логика периодических задач

Периодические задачи хранятся как правила, не как отдельные записи. Система генерирует вхождения «на лету» для отображения в календаре:

| Тип | Описание |
|-----|----------|
| Ежедневная | Каждые N дней (настраиваемый интервал) |
| Ежемесячная | Определённое число месяца |
| На конкретные даты | Список дат |
| Чётные/нечётные дни | Чётные или нечётные числа месяца |

### Управление вхождениями

- **Отмена**: можно отменить конкретное вхождение periodic задачи на определённый день
- **Редактирование**: при редактировании вхождения создаётся отдельная разовая задача на эту дату, а оригинальное вхождение отменяется
- Каждое вхождение имеет независимый статус через `periodic_exceptions`

## Допущения

1. **Обязательные теги**: три тега («отчётность», «операции», «звонок») глобальные, не удаляются и не редактируются
2. **Календарь**: отображает только задачи текущего пользователя
3. **Периодические задачи**: генерируются на лету, не хранятся как миллионы записей
4. **Неделя**: начинается с понедельника (российский стандарт)
